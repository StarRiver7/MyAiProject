# 锁（核心面试知识）

---

## 一、锁概述

> 锁是数据库用来控制并发访问的机制

### 锁的作用

```
- 保证数据一致性
- 解决并发冲突
- 实现事务隔离性
```

---

## 二、锁的分类

### 2.1 按粒度分

| 锁类型 | 粒度 | MySQL引擎 | 特点 |
|--------|------|-----------|------|
| **表锁** | 表级 | MyISAM | 开销小，冲突概率高 |
| **行锁** | 行级 | InnoDB | 开销大，冲突概率低 |
| **页锁** | 页级 | BDB | 介于表锁和行锁之间 |
| **全局锁** | 整个数据库 | 所有 | FTWRL |

### 2.2 按类型分

| 锁类型 | 简称 | 说明 |
|--------|------|------|
| **共享锁**（Shared Lock） | S锁 | 读锁，可共存 |
| **排他锁**（Exclusive Lock） | X锁 | 写锁，排斥一切 |
| **意向锁** | IX/IS | 表级锁，表示意图 |

### 2.3 兼容性矩阵

|  | S锁 | X锁 |
|--|-----|-----|
| S锁 | √兼容 | ×互斥 |
| X锁 | ×互斥 | ×互斥 |

|  | IS锁 | IX锁 |
|--|-----|-----|
| IS锁 | √兼容 | √兼容 |
| IX锁 | √兼容 | √兼容 |

## 三、表锁

### 3.1 表级锁分类

```sql
-- 1. 读锁（共享锁）
LOCK TABLE user READ;

-- 2. 写锁（排他锁）
LOCK TABLE user WRITE;

-- 3. 解锁
UNLOCK TABLES;
```

### 3.2 读锁特点

```
会话1加读锁：
- 会话1：可以读自己的表
- 会话2：可以读，但不能写（阻塞）
- 会话1：不能读其他表
```

### 3.3 写锁特点

```
会话1加写锁：
- 会话1：可以读写
- 会话2：完全阻塞（读和写都阻塞）
```

### 3.4 MyISAM表锁

```sql
-- MyISAM自动加表锁：
- 查询自动加读锁
- 更新自动加写锁

-- 表锁模式：
- READ LOCAL：并发读
- LOW_PRIORITY WRITE：降低写锁优先级
```

## 四、行锁（InnoDB）

### 4.1 行锁类型

```sql
-- 1. 共享锁（S锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;

-- 2. 排他锁（X锁）
SELECT * FROM user WHERE id = 1 FOR UPDATE;
INSERT INTO user VALUES(...);
UPDATE user SET ... WHERE id = 1;
DELETE FROM user WHERE id = 1;
```

### 4.2 锁定读

```sql
-- 共享锁：允许其他事务加S锁，不能加X锁
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;

-- 排他锁：不允许其他事务加任何锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

### 4.3 意向锁

> InnoDB自动加的表级锁，表示将对某行加锁

```sql
-- 事务A对id=1加行锁前，会先对表加IX锁
-- 事务B想加表锁，需要检查IX兼容性

IX（意向排他锁）：事务有意向对某行加X锁
IS（意向共享锁）：事务有意向对某行加S锁
```

### 4.4 记录锁（Record Lock）

> 锁定索引记录

```sql
-- 对id=1这行加锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 锁住id=1这一行
```

### 4.5 间隙锁（Gap Lock）

> 锁定索引记录之间的间隙，防止幻读

```sql
-- 假设表有id: 1, 5, 10
SELECT * FROM user WHERE id BETWEEN 3 AND 7 FOR UPDATE;
-- 锁住(1,5)和(5,10)之间的间隙
```

### 4.6 临键锁（Next-Key Lock）

> 记录锁 + 间隙锁的组合

```sql
-- 记录锁锁定已有行
-- 间隙锁锁定行之间的间隙
-- 共同构成临键锁

对于索引有：1, 5, 10
临键锁：(负无穷,1], (1,5], (5,10], (10,正无穷)
```

### 4.7 插入意向锁（Insert Intention Lock）

> INSERT操作在等待间隙锁时产生的锁

```sql
-- 场景：
-- 事务A：WHERE id > 5 AND id < 10 FOR UPDATE（持有间隙锁）
-- 事务B：INSERT id=7（需要等待插入意向锁）

-- 插入意向锁是特殊的间隙锁
-- 只阻塞同条件的INSERT，不阻塞其他操作
```

## 五、锁的算法

### 5.1 记录锁算法

> 唯一性检索（主键、唯一索引）时使用

```sql
-- 等值查询精确匹配到记录
SELECT * FROM user WHERE id = 5 FOR UPDATE;
-- 锁住id=5这一行
```

### 5.2 间隙锁算法

> 非唯一索引等值查询时使用

```sql
-- 非唯一索引查询
SELECT * FROM user WHERE age = 25 FOR UPDATE;
-- 锁定所有age=25的记录之间的间隙
```

### 5.3 临键锁算法

> 非唯一索引范围查询时使用

```sql
-- 范围查询
SELECT * FROM user WHERE age >= 20 AND age <= 30 FOR UPDATE;
-- 锁定所有满足条件的范围
```

## 六、锁问题

### 6.1 死锁（高频面试点）

> 两个或多个事务相互等待对方持有的锁

#### 死锁产生条件

```
1. 互斥：资源只能被一个事务持有
2. 持有并等待：持有资源的同时请求其他资源
3. 不可抢占：资源不能被强制释放
4. 循环等待：形成事务等待循环
```

#### 死锁示例

```
事务A：                    事务B：
BEGIN;                    BEGIN;
UPDATE user SET name='A' WHERE id=1;  -- 锁住id=1
                         UPDATE user SET name='B' WHERE id=2;  -- 锁住id=2
UPDATE user SET name='C' WHERE id=2;  -- 等待id=2锁
                         UPDATE user SET name='D' WHERE id=1;  -- 等待id=1锁
-- 死锁！
```

#### 死锁处理策略

```sql
-- 1. InnoDB自动检测死锁并回滚小事务
-- 2. 设置锁等待超时
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';  -- 默认50秒

-- 3. 主动解决死锁
-- 等待InnoDB自动回滚或手动ROLLBACK
```

#### 避免死锁

```
1. 保持事务简短
2. 按固定顺序访问表
3. 降低隔离级别
4. 添加合理索引
5. 避免大事务
```

### 6.2 脏读

> 读到其他事务未提交的数据

### 6.3 不可重复读

> 同一事务中两次读取结果不一致

### 6.4 幻读

> 同一事务中两次查询结果集不同

## 七、锁相关命令

```sql
-- 查看锁信息
SHOW STATUS LIKE 'Innodb_row_lock%';

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看当前锁
SELECT * FROM information_schema.INNODB_LOCKS;

-- 查看事务锁
SELECT * FROM information_schema.INNODB_TRX;

-- 设置死锁检测
SET GLOBAL innodb_deadlock_detect = ON;

-- 设置锁等待超时
SET GLOBAL innodb_lock_wait_timeout = 10;
```

## 八、MVCC与锁的关系

### 8.1 读操作

```
快照读（普通SELECT）：
- 不加锁
- 通过MVCC实现并发
- 可能读到历史数据

当前读（SELECT...FOR UPDATE / INSERT / UPDATE / DELETE）：
- 加行锁
- 读取最新数据
```

### 8.2 写操作

```
所有写操作都是当前读：
- 加排他锁
- 读取最新数据
- 写完后通过MVCC可见性判断
```
