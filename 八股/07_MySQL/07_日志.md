# 日志（核心面试知识）

---

## 一、MySQL日志分类

| 日志类型 | 作用 | 存储位置 |
|----------|------|----------|
| **错误日志** | 记录错误信息 | log_error |
| **查询日志** | 记录所有SQL | general_log |
| **慢查询日志** | 记录慢SQL | slow_query_log |
| **Binlog** | 记录数据变更 | binlog文件 |
| **Redo Log** | 重做日志 | ib_logfile |
| **Undo Log** | 回滚日志 | ibdata |

---

## 二、Binlog（二进制日志）- 高频面试点

### 2.1 作用

```
1. 主从复制：传递数据变更给从库
2. 数据恢复：基于时间点恢复
3. 审计：记录所有数据变更
```

### 2.2 记录内容

```sql
-- 记录内容：
- DDL（CREATE/DROP/ALTER）
- DML（INSERT/UPDATE/DELETE）
- 不记录SELECT和SHOW

-- 记录格式：
- 语句原文
- 行的变更（行模式）
- 混合模式
```

### 2.3 日志格式

```sql
-- 查看格式
SHOW VARIABLES LIKE 'binlog_format';

-- 设置格式
SET GLOBAL binlog_format = 'STATEMENT|ROW|MIXED';
```

| 格式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| STATEMENT | 记录SQL语句 | 日志小、解释清晰 | 函数/存储过程结果不确定 |
| ROW | 记录行变更 | 精确、可靠 | 日志大 |
| MIXED | 混合模式 | 自动切换 | 不确定 |

### 2.4 相关命令

```sql
-- 开启Binlog
SHOW VARIABLES LIKE 'log_bin';
SET GLOBAL log_bin = 'mysql-bin';

-- 查看Binlog文件列表
SHOW BINARY LOGS;
SHOW MASTER STATUS;

-- 查看Binlog内容
SHOW BINLOG EVENTS IN 'mysql-bin.000001';

-- 刷新日志
FLUSH LOGS;

-- 删除Binlog
PURGE BINARY LOGS TO 'mysql-bin.000010';
PURGE BINARY LOGS BEFORE '2024-01-01';
```

### 2.5 Binlog配置参数

```sql
-- 文件大小
max_binlog_size = 1GB

-- 缓存大小
binlog_cache_size = 4MB

-- 刷盘策略
sync_binlog = 1  -- 每事务刷盘
sync_binlog = 0  -- 由操作系统决定
sync_binlog = N  -- N个事务后刷盘

-- 保留天数
expire_logs_days = 7
```

## 三、Redo Log（重做日志）- 高频面试点

### 3.1 作用

```
1. 实现事务持久性
2. 崩溃恢复
3. 提高写入性能（ WAL：Write Ahead Logging）
```

### 3.2 工作原理

```
写入流程：
1. 事务开始
2. 先写Redo Log（Prepare状态）
3. 写数据页到Buffer Pool
4. 事务提交（Redo Log变Commit状态）
5. 后续异步刷盘

崩溃恢复：
- Redo Log是顺序I/O，性能高
- 通过Redo Log恢复未刷盘的数据
```

### 3.3 与Binlog的区别

| 对比项 | Redo Log | Binlog |
|--------|----------|--------|
| 位置 | InnoDB特有 | Server层通用 |
| 内容 | 物理修改（页操作） | 逻辑修改（SQL语句） |
| 格式 | 循环写入 | 追加写入 |
| 用途 | 崩溃恢复 | 主从复制/恢复 |
| 刷盘 | 事务提交时 | 由sync_binlog决定 |

### 3.4 配置参数

```sql
-- 查看Redo Log
SHOW VARIABLES LIKE 'innodb_log%';

-- 日志文件大小
innodb_log_file_size = 48MB

-- 日志组文件数
innodb_log_files_in_group = 3

-- 刷盘策略
innodb_flush_log_at_trx_commit = 1
-- 1：事务提交时刷盘（最安全）
-- 0：每秒刷盘
-- 2：写缓冲区，由操作系统刷盘
```

## 四、Undo Log（回滚日志）

### 4.1 作用

```
1. 实现事务原子性（回滚）
2. 实现MVCC（版本链）
```

### 4.2 存储位置

```sql
-- 存储在共享表空间（ibdata）
-- 或者独立Undo表空间（MySQL 5.6+）

-- 查看Undo表空间
SHOW VARIABLES LIKE 'innodb_undo_directory';
SHOW VARIABLES LIKE 'innodb_undo_tablespaces';
```

### 4.3 与Redo Log的关系

```
Redo Log：记录"做什么"
Undo Log：记录"怎么恢复"

关系：
- Undo Log需要Redo Log保护（崩溃恢复时Undo Log也需要恢复）
- 两者配合实现事务的ACID特性
```

## 七、日志与事务关系

### 7.1 两阶段提交

```
准备阶段（Prepare）：
1. 写Redo Log（Prepare）
2. 写Binlog

提交阶段（Commit）：
3. 写Redo Log（Commit）

保证：
- 两者一致性
- 崩溃后可恢复
```

### 7.2 崩溃恢复流程

```
崩溃发生
   ↓
检查Redo Log状态：
   ↓
   ├─ Redo Log完整 + Commit → 提交
   │
   ├─ Redo Log完整 + Prepare → 检查Binlog：
   │     ├─ Binlog有记录 → 提交
   │     └─ Binlog无记录 → 回滚
   │
   └─ Redo Log不完整 → 回滚
```
