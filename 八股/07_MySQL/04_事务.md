## 一、事务概述

> 事务是数据库操作的**基本单位**，要么全部成功，要么全部失败

### 事务的特性（ACID）- 面试必问

| 特性 | 说明 | 原理 |
|------|------|------|
| **A**tomicity | 原子性 | Undo Log回滚 |
| **C**onsistency | 一致性 | 约束规则 |
| **I**solation | 隔离性 | 锁 + MVCC |
| **D**urability | 持久性 | Redo Log刷盘 |

## 二、四种隔离级别（高频面试点）

### 2.1 隔离级别定义

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| READ UNCOMMITTED | √可能 | √可能 | √可能 |
| READ COMMITTED | × | √可能 | √可能 |
| REPEATABLE READ（MySQL默认） | × | × | √可能 |
| SERIALIZABLE | × | × | × |

### 2.2 各隔离级别详解

```sql
-- 设置隔离级别
SET [SESSION|GLOBAL] TRANSACTION ISOLATION LEVEL 级别;

-- 查看当前隔离级别
SELECT @@transaction_isolation;
```

#### READ UNCOMMITTED（读未提交）

```
特点：可以读到其他事务未提交的数据
问题：脏读

时间线：
T1: 事务A开始
T2: 事务A读 id=1 → 值为100
T3: 事务B开始
T4: 事务B更新 id=1 → 值为200（未提交）
T5: 事务A读 id=1 → 读到200（脏数据）
T6: 事务B回滚
T7: 事务A读 id=1 → 又回到100
```

#### READ COMMITTED（读已提交）

```
特点：只能读到已提交的数据
问题：不可重复读

时间线：
T1: 事务A读 id=1 → 100
T2: 事务B更新 id=1 → 200（已提交）
T3: 事务A再读 id=1 → 200（不可重复读）
```

#### REPEATABLE READ（可重复读）- MySQL默认

```
特点：同一个事务中多次读取结果一致
问题：幻读（InnoDB通过MVCC+间隙锁解决）

时间线：
T1: 事务A读 id=1 → 100
T2: 事务B插入 id=2 → 200（已提交）
T3: 事务A再读 id=1 → 100（不变）
T4: 事务A插入 id=2 → 成功（幻读感知到）
```

#### SERIALIZABLE（串行化）

```
特点：完全串行执行
问题：性能最差

实现：
- 读锁：SELECT → LOCK IN SHARE MODE
- 写锁：INSERT/UPDATE/DELETE → 加写锁
- 阻塞其他事务的读写操作
```

## 三、事务隔离级别实现原理

### 3.1 MVCC（多版本并发控制）

> 核心思想：读操作不加锁，通过版本链实现快照读

#### 隐藏字段

每行数据有两个隐藏列：
- `DB_TRX_ID`：最近修改的事务ID
- `DB_ROLL_PTR`：指向undo log的指针

#### 版本链

```
每次修改都会写入undo log，形成版本链：

undo log:
    [id=1, value=100, DB_ROLL_PTR=null]
    ↓
    [id=1, value=200, DB_ROLL_PTR=指针1]
    ↓
    [id=1, value=300, DB_ROLL_PTR=指针2]

读取时：根据事务ID和undo log构建快照
```

#### ReadView（读视图）

> 记录当前活跃事务ID列表和最小/最大事务ID

```sql
-- 不同隔离级别的ReadView生成时机：

READ COMMITTED：
- 每次SELECT都生成新的ReadView
- 能读到已提交事务的修改

REPEATABLE READ：
- 事务第一次SELECT时生成ReadView
- 整个事务复用同一个ReadView
- 保证可重复读
```

### 3.2 当前读 vs 快照读

```sql
-- 快照读：普通SELECT，不加锁
SELECT * FROM user WHERE id = 1;  -- 快照读

-- 当前读：读取最新数据，加锁
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;  -- 共享锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;           -- 排他锁
INSERT INTO user VALUES(...);   -- 排他锁
UPDATE user SET ...;            -- 排他锁
DELETE FROM user WHERE ...;     -- 排他锁
```

## 四、事务相关SQL

```sql
-- 开启事务
START TRANSACTION;  或  BEGIN;

-- 提交事务
COMMIT;

-- 回滚事务
ROLLBACK;

-- 设置保存点
SAVEPOINT point1;

-- 回滚到保存点
ROLLBACK TO SAVEPOINT point1;

-- 设置事务自动提交（默认ON）
SET autocommit = 0;  -- 关闭自动提交
SET autocommit = 1;  -- 开启自动提交

-- 查看自动提交状态
SELECT @@autocommit;
```

## 五、分布式事务

### 5.1 CAP定理

```
C (Consistency)：一致性
A (Availability)：可用性
P (Partition tolerance)：分区容错性

三者不可兼得，最多同时满足两个：
- CA：传统单机数据库
- CP：Redis、HBase、MongoDB
- AP：Eureka、Cassandra
```

### 5.2 BASE理论

```
Basically Available：基本可用
Soft state：软状态
Eventually consistent：最终一致性
```

### 5.3 分布式事务解决方案

#### XA事务（两阶段提交）

```
阶段1：预提交（Prepare）
- 各参与者执行本地事务但不提交
- 返回成功/失败

阶段2：提交/回滚（Commit/Rollback）
- 协调者根据结果决定提交或回滚

问题：同步阻塞、单点故障、数据不一致
```

#### TCC（Try-Confirm-Cancel）

```
Try：预留资源
Confirm：确认执行
Cancel：取消回滚

适用场景：高性能要求的分布式事务
```

## 六、事务并发问题

### 6.1 脏读

> 读取到其他事务未提交的数据

```
事务A读取了事务B未提交的数据，然后事务B回滚
```

### 6.2 不可重复读

> 同一事务中，两次读取同一数据结果不同

```
事务A读取数据→事务B修改并提交→事务A再读→结果不同
```

### 6.3 幻读

> 同一事务中，两次查询结果集不同（多了或少了行）

```
事务A查询数据→事务B插入新行并提交→事务A再查→结果集多了行
```
