# 索引（核心面试知识）

---

## 一、索引概述

> 索引是帮助MySQL高效获取数据的**数据结构**

### 索引的本质

```
索引 = 数据结构 + 有序排列 + 快速查找
```

### 索引的优缺点

| 优点 | 缺点 |
|------|------|
| 提高查询速度 | 占用磁盘空间 |
| 减少I/O次数 | 降低写入速度（INSERT/UPDATE/DELETE） |
| 加速排序 | 索引维护成本 |

---

## 二、索引数据结构（必须掌握）

### 2.1 B+ Tree索引（MySQL默认）

```
B+ Tree 结构：
                    [15]
              /              \
         [5,10]              [20,25]
        /      \             /        \
     [1,3]  [5,7] [8,10]  [15,18] [20,22] [25,30]

特点：
- 叶子节点存储所有数据（卫星指针）
- 叶子节点之间用双向链表连接
- 层级更低，I/O次数更少
```

### 2.2 Hash索引

```
Hash结构：
Key → Hash函数 → 桶地址

特点：
- 等值查询极快
- 无法范围查询
- 无法排序
- 存在Hash冲突
```

### 2.3 对比

| 特性 | B+ Tree | Hash |
|------|---------|------|
| 等值查询 | √ | √√ |
| 范围查询 | √ | × |
| 排序 | √ | × |
| 前缀匹配 | √ | × |
| 复合索引 | √ | × |
| Hash冲突 | - | 存在 |

## 三、索引分类

### 3.1 按数据结构分

| 类型 | 说明 |
|------|------|
| B+ Tree索引 | 默认，常用 |
| Hash索引 | MEMORY引擎支持 |
| R-Tree索引 | 空间数据索引 |
| Full-text索引 | 全文检索 |

### 3.2 按存储方式分

| 类型 | 说明 |
|------|------|
| 聚簇索引 | 数据和索引在一起，葉子节点存数据 |
| 非聚簇索引 | 数据和索引分开，叶子节点存主键 |

### 3.3 按字段特性分

| 类型 | 说明 |
|------|------|
| 主键索引 | 主键列，值唯一且非空 |
| 唯一索引 | 值唯一，可为空 |
| 普通索引 | 普通字段，无限制 |
| 前缀索引 | 字段前N个字符 |

### 3.4 按字段个数分

| 类型 | 说明 |
|------|------|
| 单列索引 | 一个字段 |
| 复合索引 | 多个字段 |

## 四、聚簇索引 vs 非聚簇索引（高频面试点）

### 4.1 核心区别

```
聚簇索引（Clustered Index）：
- 叶子节点存储完整数据行
- 每表只有一个聚簇索引（主键）
- 数据物理存储顺序与索引顺序一致

非聚簇索引（Secondary Index）：
- 叶子节点存储主键值
- 可能存在多个非聚簇索引
- 先查索引再回表查数据
```

### 4.2 回表查询

```sql
-- 创建非聚簇索引
CREATE INDEX idx_name ON user(name);

-- 执行查询
SELECT * FROM user WHERE name = '张三';

-- 执行过程（回表）：
1. 先在idx_name索引树找到name='张三'的叶子节点
2. 叶子节点存储的是主键ID
3. 再拿主键ID去聚簇索引查找完整数据
4. 这一步称为：回表
```

### 4.3 覆盖索引（索引覆盖）

```sql
-- 索引覆盖：不需要回表
SELECT name FROM user WHERE name = '张三';
-- idx_name索引树叶子节点直接有name值，无需回表

-- 非覆盖索引：需要回表
SELECT * FROM user WHERE name = '张三';  -- 需回表取所有字段
```

## 五、复合索引（高频面试点）

### 5.1 复合索引结构

```sql
-- 创建复合索引
CREATE INDEX idx_name_age ON user(name, age);

-- 等价于创建了三个索引：
- (name)
- (name, age)
- (name, age, email)  -- 如果有email的话
```

### 5.2 最左前缀原则

> 复合索引从最左开始匹配，断则失效

```sql
-- 假设索引：idx(name, age, salary)

-- 能命中索引的情况：
SELECT * FROM user WHERE name = '张三';           -- √ 命中name
SELECT * FROM user WHERE name = '张三' AND age = 25;  -- √ 命中name,age
SELECT * FROM user WHERE name = '张三' AND age = 25 AND salary = 5000;  -- √ 全部命中

-- 不能命中索引的情况：
SELECT * FROM user WHERE age = 25;                -- × 不满足最左前缀
SELECT * FROM user WHERE salary = 5000;            -- × 不满足最左前缀
SELECT * FROM user WHERE age = 25 AND name = '张三';  -- √ 优化器会调整顺序

-- 部分命中：
SELECT * FROM user WHERE name = '张三' AND salary = 5000;  -- △ 部分命中（只用到name）
```

### 5.3 索引下推（ICP）

```sql
-- 假设索引：idx(name, age)
SELECT * FROM user WHERE name = '张三' AND age > 20;

-- 不使用ICP：
1. 先用name='张三'筛选出所有记录
2. 再在MySQL Server层过滤age>20

-- 使用ICP（MySQL 5.6+）：
1. 在索引树中同时用name='张三'和age>20筛选
2. 只返回匹配条件的索引记录
3. 减少回表次数
```

## 六、索引失效场景（高频面试点）

### 6.1 导致索引失效的情况

```sql
-- 1. 使用函数或运算
SELECT * FROM user WHERE YEAR(create_time) = 2024;  -- ×
SELECT * FROM user WHERE create_time + 1 = '2024-01-01';  -- ×

-- 2. 类型转换
SELECT * FROM user WHERE name = 123;  -- × name是varchar，123被转成'123'

-- 3. LIKE以%开头
SELECT * FROM user WHERE name LIKE '%张%';  -- ×
SELECT * FROM user WHERE name LIKE '张%';   -- √

-- 4. OR前后不全是索引列
SELECT * FROM user WHERE name = '张三' OR age = 25;  -- ×（age无索引）

-- 5. NOT IN / NOT EXISTS
SELECT * FROM user WHERE id NOT IN (1,2,3);  -- △ 可能失效
SELECT * FROM user WHERE name NOT IN ('张三','李四');  -- △ 可能失效

-- 6. 负向查询（部分）
SELECT * FROM user WHERE age != 25;   -- △ 可能失效
SELECT * FROM user WHERE age NOT IN (20,25,30);  -- △ 可能失效

-- 7. 复合索引不满足最左前缀
-- 假设索引：idx(name, age, salary)
SELECT * FROM user WHERE age = 25;  -- ×
```

### 6.2 保持索引有效的原则

> - 避免在索引列上使用函数
> - 避免类型转换
> - LIKE前缀要精确
> - OR条件要全索引
> - 遵循最左前缀原则
> - 尽量使用覆盖索引

## 七、索引创建原则（高频面试点）

### 7.1 创建索引的场景

| 场景 | 原因 |
|------|------|
| 主键自动建立主键索引 | 唯一且非空 |
| 频繁查询的字段 | 减少I/O |
| 频繁排序的字段 | 避免filesort |
| 频繁GROUP BY的字段 | 减少扫描 |
| 频繁WHERE条件的字段 | 加速筛选 |
| 复合索引替代单列索引 | 减少索引数量 |

### 7.2 不创建索引的场景

| 场景 | 原因 |
|------|------|
| 重复度高的字段 | 区分度低 |
| 很少使用的字段 | 浪费空间 |
| 频繁更新的字段 | 维护成本高 |
| TEXT/BLOB类型 | 占用空间大 |

## 八、Hint索引强制使用

```sql
-- 强制使用某个索引
SELECT * FROM user USE INDEX(idx_name) WHERE name = '张三';

-- 强制忽略某个索引
SELECT * FROM user IGNORE INDEX(idx_name) WHERE name = '张三';

-- 强制使用某个执行计划
SELECT * FROM user FORCE INDEX(idx_name) WHERE name = '张三';
```
