# 主从复制（核心面试知识）

---

## 一、主从复制概述

### 1.1 作用

```
1. 读写分离：主库写，从库读
2. 数据备份：从库作为备份
3. 负载均衡：分散查询压力
4. 高可用：故障转移
```

### 1.2 基本原理

```
主库：
1. 接收客户端写请求
2. 执行写操作
3. 写Binlog
4. 返回客户端

从库：
1. I/O线程：拉取主库Binlog
2. 写入Relay Log（中继日志）
3. SQL线程：执行Relay Log
```

## 二、主从复制原理（高频面试点）

### 2.1 复制流程

```
┌─────────┐     Binlog      ┌─────────┐     Relay Log     ┌─────────┐
│  主库   │ ──────────────→ │  从库   │ ←──────────────── │ 从库    │
│         │                 │ I/O线程 │                   │ SQL线程 │
└─────────┘                 └─────────┘                   └─────────┘
```

```
详细步骤：
1. 主库记录Binlog（记录数据变更）
2. 从库I/O线程连接主库，请求Binlog
3. 主库BinlogDump线程发送Binlog给从库
4. 从库I/O线程接收，写入Relay Log
5. 从库SQL线程读取Relay Log
6. 从库SQL线程执行SQL（重放）
```

### 2.2 三个线程

| 线程 | 位置 | 作用 |
|------|------|------|
| BinlogDump | 主库 | 发送Binlog给从库 |
| I/O线程 | 从库 | 拉取Binlog写入Relay Log |
| SQL线程 | 从库 | 执行Relay Log |

### 2.3 复制方式

#### 异步复制

```
主库提交事务后，立即返回客户端
不关心从库是否接收

特点：
- 延迟大
- 可能丢数据
- 性能最好
```

#### 半同步复制

```
主库提交事务后，等待至少一个从库接收成功
才返回客户端

特点：
- 延迟较小
- 数据不易丢
- 性能适中

参数：rpl_semi_sync_master_wait_point
- AFTER_COMMIT：提交后等待
- AFTER_SYNC：从库写入后等待（5.7+）
```

#### 全同步复制

```
主库提交事务后，等待所有从库执行完成
才返回客户端

特点：
- 无延迟
- 不丢数据
- 性能最差
```

## 三、主从复制配置

### 3.1 主库配置

```ini
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
expire_logs_days = 7
```

```sql
-- 创建复制账号
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 获取Binlog位置
SHOW MASTER STATUS;
-- 记录：File和Position
```

### 3.2 从库配置

```ini
[mysqld]
server-id = 2
relay-log = relay-bin
read_only = ON
```

```sql
-- 设置主库连接
CHANGE MASTER TO
    MASTER_HOST = '主库IP',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_LOG_FILE = 'mysql-bin.000001',
    MASTER_LOG_POS = 157;

-- 启动复制
START SLAVE;

-- 查看状态
SHOW SLAVE STATUS\G
```

### 3.3 关键参数

```sql
-- 主库
server-id               -- 唯一ID
log-bin                 -- 开启Binlog
binlog_format           -- Binlog格式

-- 从库
server-id               -- 唯一ID
relay-log               -- 中继日志
read-only               -- 只读
```

## 四、主从复制问题

### 4.1 复制延迟

```
原因：
1. 从库配置低
2. 大事务（一个事务太大）
3. 从库压力大（查询太多）
4. 网络延迟
5. 单线程复制（传统模式）

解决方案：
1. 升级硬件
2. 拆分大事务
3. 减少从库查询压力
4. 升级到并行复制
5. 使用推荐配置
```

### 4.2 数据不一致

```
原因：
1. 复制中断
2. 错误执行SQL
3. 主从配置不同
4. 软删除与硬删除冲突

解决方案：
1. 定期校验数据
2. 开启GTID模式
3. 添加复制过滤规则
```

### 4.3 主库故障

```
问题：需要手动切换主库

解决方案：
1. 使用MHA/Orchestrator自动切换
2. 配置半同步复制
3. 使用Orchestrator可视化工具
```

## 五、GTID复制（MySQL 5.7+）

### 5.1 GTID概述

> Global Transaction Identifier：全局事务ID

```
格式：server_uuid:transaction_id

例如：3E11FA47-71CA-11E1-9E33-C80AA9429562:23
```

### 5.2 GTID优势

```
1. 自动找位点：不需要手动指定Position
2. 简单切换：主从切换更简单
3. 一致性更强：基于事务追踪
4. 故障恢复快：自动定位位点
```

### 5.3 GTID配置

```ini
# 主库
gtid_mode = ON
enforce_gtid_consistency = ON

# 从库
gtid_mode = ON
enforce_gtid_consistency = ON
relay_log_recovery = ON
```

### 5.4 GTID工作流程

```
主库：
1. 事务提交时生成GTID
2. 写入Binlog
3. 返回GTID给客户端

从库：
1. 接收Binlog（包含GTID）
2. 检测本地执行过的GTID集合
3. 只执行未执行过的事务
```

## 六、并行复制

### 6.1 问题

```
传统复制：SQL线程单线程执行
问题：大事务延迟严重

解决：多线程并行复制
```

### 6.2 并行复制策略

```sql
-- 5.6：库级别并行
slave_parallel_workers = 4

-- 5.7：组提交并行（基于Commit_parent）
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 8

-- 8.0：writeset并行
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 16
```

## 七、主从复制过滤

### 7.1 复制过滤规则

```sql
-- 主库：只记录某些库到Binlog
binlog-do-db = database_name
binlog-ignore-db = database_name

-- 从库：只应用某些库/表
replicate_do_db = database_name
replicate_ignore_db = database_name
replicate_do_table = database.table
replicate_ignore_table = database.table
```

### 7.2 注意事项

```
1. 过滤规则基于默认数据库
2. 跨库操作需要小心
3. 建议在主库过滤，减少Binlog大小
```

## 八、读写分离

### 8.1 实现方式

```
1. 应用层实现：代码中判断读写
2. 代理层实现：MySQL Proxy/Atlas/ProxySQL
3. 组件实现：ShardingSphere
```

### 8.2 路由策略

```
根据语句类型路由：
- INSERT/UPDATE/DELETE → 主库
- SELECT → 从库

根据业务需求强制路由：
- 读主库：SELECT * FROM ... MASTER
- 读从库：SELECT * FROM ... SLAVE
```
