# 哨兵模式（核心面试知识）

---

## 一、哨兵模式概述

> Redis Sentinel（哨兵）是Redis的高可用解决方案，用于监控Redis主从集群的健康状态

### 1.1 作用

```
1. 监控：监控主库和从库的健康状态
2. 自动故障转移：主库故障时自动将从库升级为主库
3. 通知：当监控的实例发生变化时通知客户端
4. 配置管理：自动更新客户端的主库信息
```

---

## 二、哨兵工作原理

### 2.1 哨兵的结构

```
┌─────────────┐
│  哨兵集群   │
└─────┬───────┘
      │ 监控
┌─────▼───────┐     复制    ┌─────────────┐
│  主库(6379)  │ ─────────> │  从库(6380)  │
└─────────────┘            └─────────────┘
                             └─────────────┘
                                  ▲
                                  │
┌─────────────┐     复制    ┌─────┴───────┐
│  主库(6379)  │ ─────────> │  从库(6381)  │
└─────────────┘            └─────────────┘
```

### 2.2 哨兵的三个阶段

#### 1. 监控阶段

```
- 哨兵定期向主库和从库发送PING命令
- 检测实例是否在线
- 收集主从状态信息
```

#### 2. 通知阶段

```
- 哨兵之间交换信息
- 共享主从状态
- 选举领导者哨兵
```

#### 3. 故障转移阶段

```
- 确认主库故障
- 选举新的主库
- 其他从库指向新主库
- 通知客户端
```

## 三、哨兵配置

### 3.1 配置文件

```bash
# 哨兵配置文件（sentinel.conf）

# 哨兵端口
port 26379

# 哨兵ID
sentinel myid c928a019b6a8f97a08c112c8a34a05a075a3f8f9

# 监控主库
sentinel monitor mymaster 127.0.0.1 6379 2
# 格式：sentinel monitor <主库名称> <主库IP> <主库端口> <需要多少个哨兵同意故障转移>

# 主库密码
sentinel auth-pass mymaster your_password

# 故障转移超时时间
sentinel down-after-milliseconds mymaster 30000  # 30秒

# 故障转移超时时间
sentinel failover-timeout mymaster 180000  # 180秒

# 并行同步数量
sentinel parallel-syncs mymaster 1
```

### 3.2 启动哨兵

```bash
# 启动哨兵
redis-sentinel /path/to/sentinel.conf

# 或者使用redis-server启动
redis-server /path/to/sentinel.conf --sentinel

# 查看哨兵状态
redis-cli -p 26379 info sentinel
```

## 四、故障转移流程

### 4.1 故障检测

```
1. 哨兵向主库发送PING命令
2. 如果主库在down-after-milliseconds时间内没有响应
3. 哨兵标记主库为S_DOWN（主观下线）
4. 哨兵向其他哨兵发送询问
5. 如果超过quorum个哨兵认为主库下线
6. 标记主库为O_DOWN（客观下线）
```

### 4.2 故障转移

```
1. 选举领导者哨兵
2. 领导者哨兵选择新主库：
   - 过滤不健康的从库
   - 选择slave-priority最高的从库
   - 选择复制偏移量最大的从库
   - 选择runid最小的从库
3. 执行故障转移：
   - 向新主库发送SLAVEOF no one
   - 向其他从库发送SLAVEOF新主库IP端口
   - 更新配置
4. 通知客户端
```

## 五、哨兵集群

### 5.1 哨兵数量

```
- 建议3-5个哨兵
- 奇数个哨兵，避免脑裂
- 分布在不同的机器上
```

### 5.2 哨兵之间的通信

```
- 哨兵之间通过Gossip协议通信
- 定期交换主从状态信息
- 选举领导者哨兵
```

### 5.3 哨兵的监控范围

```
- 一个哨兵可以监控多个主库
- 多个哨兵可以监控同一个主库
- 形成多对多的监控关系
```

## 六、监控和管理

### 6.1 查看哨兵状态

```bash
# 查看哨兵信息
redis-cli -p 26379 info sentinel

# 查看主库状态
redis-cli -p 26379 sentinel master mymaster

# 查看从库状态
redis-cli -p 26379 sentinel slaves mymaster

# 查看其他哨兵
redis-cli -p 26379 sentinel sentinels mymaster
```

### 6.2 手动故障转移

```bash
# 手动触发故障转移
redis-cli -p 26379 sentinel failover mymaster
```

### 6.3 哨兵命令

| 命令 | 说明 |
|------|------|
| sentinel masters | 查看所有监控的主库 |
| sentinel slaves <master> | 查看指定主库的从库 |
| sentinel sentinels <master> | 查看指定主库的哨兵 |
| sentinel get-master-addr-by-name <master> | 获取主库地址 |
| sentinel failover <master> | 手动故障转移 |
| sentinel reset <pattern> | 重置哨兵状态 |

## 七、哨兵常见问题

### 7.1 脑裂

```
原因：
- 网络分区导致哨兵认为主库下线
- 主库实际仍在运行
- 哨兵选举新主库，形成两个主库

解决方案：
- 合理设置quorum
- 使用redis.conf的min-replicas-to-write参数
- 监控哨兵状态
```

### 7.2 故障转移失败

```
原因：
- 所有从库都不可用
- 网络问题
- 哨兵配置错误

解决方案：
- 确保至少有一个健康的从库
- 检查网络连接
- 验证哨兵配置
```

### 7.3 客户端连接问题

```
原因：
- 客户端不知道主库已切换
- 缓存了旧的主库地址

解决方案：
- 使用哨兵提供的服务发现
- 定期检查主库地址
- 使用Redis客户端的哨兵模式
```

## 八、哨兵最佳实践

### 8.1 部署建议

```
1. 至少3个哨兵，分布在不同机器
2. 哨兵和Redis实例分离部署
3. 合理设置故障检测时间
4. 监控哨兵状态
```

### 8.2 配置建议

```bash
# 哨兵配置
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
sentinel parallel-syncs mymaster 1

# 主库配置
min-replicas-to-write 1
min-replicas-max-lag 10
```

### 8.3 客户端使用

```python
# Python示例（使用redis-py）
from redis.sentinel import Sentinel

# 哨兵列表
sentinel = Sentinel([
    ('127.0.0.1', 26379),
    ('127.0.0.1', 26380),
    ('127.0.0.1', 26381)
], socket_timeout=0.1)

# 获取主库连接
master = sentinel.master_for('mymaster', socket_timeout=0.1)

# 获取从库连接
slave = sentinel.slave_for('mymaster', socket_timeout=0.1)

# 使用
master.set('key', 'value')
slave.get('key')
```
