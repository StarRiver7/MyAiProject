# 持久化（核心面试知识）

---

## 一、持久化概述

> Redis持久化是将内存中的数据保存到磁盘，防止数据丢失

### 1.1 持久化方式

| 方式 | 说明 |
|------|------|
| **RDB** | 快照方式，定期将内存数据保存到磁盘 |
| **AOF** | 追加方式，记录所有写操作命令 |

---

## 二、RDB（Redis Database）

### 2.1 原理

```
1. Redis创建子进程
2. 子进程将内存数据写入临时文件
3. 写入完成后替换旧的RDB文件
4. 父进程继续处理请求

优点：
- 紧凑的二进制文件
- 恢复速度快
- 适合备份

缺点：
- 可能丢失数据
- 生成快照时阻塞
```

### 2.2 配置

```bash
# 配置文件

# RDB文件名称
dbfilename dump.rdb

# RDB文件保存路径
dir /var/lib/redis

# 触发条件
save 900 1      # 900秒内至少1个key变化
save 300 10     # 300秒内至少10个key变化
save 60 10000   # 60秒内至少10000个key变化

# 压缩
rdbcompression yes  # 压缩RDB文件

# 校验
rdbchecksum yes     # 校验RDB文件
```

### 2.3 手动触发

```bash
# 同步保存（阻塞）
save

# 异步保存（非阻塞）
bgsave

# 查看RDB状态
info persistence
```

### 2.4 恢复

```bash
# Redis启动时自动加载RDB文件
# 启动时会检查dir目录下的dump.rdb文件
```

## 三、AOF（Append Only File）

### 3.1 原理

```
1. 记录所有写操作命令
2. 以追加方式写入AOF文件
3. 恢复时重新执行这些命令

优点：
- 数据安全性高
- 可配置刷盘策略
- 可读性好

缺点：
- 文件体积大
- 恢复速度慢
- 写性能稍低
```

### 3.2 配置

```bash
# 配置文件

# 开启AOF
appendonly yes

# AOF文件名称
appendfilename "appendonly.aof"

# 刷盘策略
appendfsync always   # 每次写操作都刷盘（最安全）
appendfsync everysec # 每秒刷盘（推荐）
appendfsync no       # 由操作系统决定（性能最好）

# 重写配置
auto-aof-rewrite-percentage 100  # 增长100%时重写
auto-aof-rewrite-min-size 64mb   # 最小64MB时重写
```

### 3.3 重写（Rewrite）

```
重写目的：
- 压缩文件大小
- 移除过期键
- 合并重复命令

重写过程：
1. 创建子进程
2. 子进程读取内存数据，生成新的AOF文件
3. 父进程将新的写操作追加到内存缓冲区
4. 子进程完成后，父进程将缓冲区内容写入新文件
5. 替换旧的AOF文件
```

### 3.4 手动触发

```bash
# 重写AOF文件
BGREWRITEAOF

# 查看AOF状态
info persistence
```

### 3.5 恢复

```bash
# Redis启动时优先加载AOF文件
# 如果AOF文件存在，使用AOF恢复
# 如果AOF文件不存在，使用RDB恢复
```

## 四、RDB vs AOF 对比

| 对比项 | RDB | AOF |
|--------|-----|-----|
| **文件大小** | 小 | 大 |
| **恢复速度** | 快 | 慢 |
| **数据安全性** | 可能丢失 | 更安全 |
| **写性能** | 高 | 稍低 |
| **可读性** | 二进制 | 文本 |
| **备份** | 适合 | 不适合 |
| **重写** | 自动 | 自动 |

## 五、混合持久化（Redis 4.0+）

### 5.1 原理

```
结合RDB和AOF的优点：
- 开头是RDB格式（快速恢复）
- 后面是AOF格式（数据安全）

配置：
aof-use-rdb-preamble yes
```

### 5.2 优点

```
1. 恢复速度快（RDB部分）
2. 数据安全性高（AOF部分）
3. 文件大小适中
```

## 六、持久化配置最佳实践

### 6.1 生产环境配置

```bash
# RDB配置
save 3600 1      # 1小时1个key变化
save 300 100     # 5分钟100个key变化
save 60 10000    # 1分钟10000个key变化

# AOF配置
appendonly yes
appendfsync everysec

# 混合持久化
aof-use-rdb-preamble yes

# 重写配置
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 6.2 备份策略

```
1. 定期备份RDB文件
2. 保存不同版本的备份
3. 异地存储备份
4. 定期测试恢复
```

## 七、持久化常见问题

### 7.1 RDB问题

```
1. 生成快照时阻塞
   - 解决方案：使用bgsave

2. 数据丢失
   - 解决方案：结合AOF

3. 大内存服务器生成RDB慢
   - 解决方案：合理设置触发条件
```

### 7.2 AOF问题

```
1. 文件过大
   - 解决方案：开启自动重写

2. 恢复速度慢
   - 解决方案：使用混合持久化

3. 刷盘策略影响性能
   - 解决方案：选择everysec
```
