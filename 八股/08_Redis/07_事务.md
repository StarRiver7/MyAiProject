# 事务（核心面试知识）

---

## 一、Redis事务概述

> Redis事务是一组命令的集合，要么全部执行，要么全部不执行

### 1.1 特点

```
1. 原子性：事务中的命令要么全部执行，要么全部不执行
2. 顺序性：事务中的命令按顺序执行，不会被其他命令插入
3. 隔离性：事务执行过程中，其他客户端看不到中间结果
4. 持久性：取决于持久化配置
```

---

## 二、事务命令

### 2.1 基本命令

| 命令 | 说明 |
|------|------|
| MULTI | 开始事务 |
| EXEC | 执行事务 |
| DISCARD | 取消事务 |
| WATCH | 监视键，用于乐观锁 |
| UNWATCH | 取消监视 |

### 2.2 使用示例

```bash
# 基本事务
MULTI
SET key1 value1
SET key2 value2
GET key1
EXEC

# 取消事务
MULTI
SET key1 value1
DISCARD

# 带WATCH的事务
WATCH key1
MULTI
SET key1 value2
EXEC
```

---

## 三、事务执行流程

```
1. 客户端发送MULTI命令
2. Redis返回OK
3. 客户端发送多个命令
4. Redis将这些命令放入队列
5. 客户端发送EXEC命令
6. Redis执行队列中的所有命令
7. Redis返回执行结果
```

## 四、事务的原子性

### 4.1 命令执行失败

```
- 如果事务中的命令语法错误，整个事务会被拒绝执行
- 如果事务中的命令在执行时失败（如类型错误），其他命令仍会执行
- Redis事务不支持回滚
```

### 4.2 示例

```bash
# 语法错误（整个事务失败）
MULTI
SET key1 value1
INVALID_COMMAND
EXEC  # 会失败

# 执行错误（部分执行）
MULTI
SET key1 value1
INCR key1  # 类型错误
SET key2 value2
EXEC  # key2会被设置
```

## 五、WATCH命令（乐观锁）

### 5.1 原理

```
- WATCH命令用于监视一个或多个键
- 如果在事务执行前，被监视的键被其他客户端修改
- 事务会被取消执行
- 用于实现乐观锁
```

### 5.2 示例

```bash
# 实现计数器的原子操作
WATCH counter
val = GET counter
val = val + 1
MULTI
SET counter $val
EXEC

# 如果counter被其他客户端修改，EXEC会返回nil
```

### 5.3 常见使用场景

```
- 并发操作同一数据
- 避免竞态条件
- 实现CAS（Compare And Swap）操作
```

## 六、事务与Lua脚本

### 6.1 Lua脚本的优势

```
- 原子执行：整个脚本作为一个原子操作
- 减少网络开销：一次发送多个命令
- 灵活性：可以编写复杂逻辑
```

### 6.2 使用示例

```bash
# 执行Lua脚本
EVAL "return redis.call('set', KEYS[1], ARGV[1])" 1 key value

# 缓存脚本
SCRIPT LOAD "return redis.call('set', KEYS[1], ARGV[1])"
# 返回脚本SHA1

# 执行缓存的脚本
EVALSHA <script_sha1> 1 key value
```

### 6.3 与事务的对比

| 对比项 | 事务 | Lua脚本 |
|--------|------|---------|
| 原子性 | 部分支持 | 完全支持 |
| 复杂度 | 简单 | 复杂 |
| 网络开销 | 多次网络往返 | 一次网络往返 |
| 灵活性 | 有限 | 高度灵活 |

## 七、事务的限制

### 7.1 不支持回滚

```
- Redis事务不支持回滚
- 原因：
  1. Redis命令执行错误通常是编程错误
  2. 简化Redis内部实现
  3. 提高性能
```

### 7.2 阻塞问题

```
- 事务执行过程中，其他命令会被阻塞
- 长时间的事务会影响性能
- 建议：
  1. 事务尽量简短
  2. 避免在事务中执行耗时操作
```

### 7.3 集群限制

```
- 事务中的命令必须在同一个槽位
- 否则会报错
- 解决方案：使用哈希标签
```

## 八、事务最佳实践

### 8.1 使用建议

```
1. 事务命令要简短
2. 避免在事务中执行耗时操作
3. 使用WATCH实现乐观锁
4. 考虑使用Lua脚本替代复杂事务
5. 合理处理EXEC返回的结果
```

### 8.2 错误处理

```
- 检查EXEC返回的结果
- 处理nil返回值（WATCH触发）
- 处理命令执行错误
```

### 8.3 性能考虑

```
- 事务会阻塞其他命令
- 长事务会影响吞吐量
- 建议：
  1. 将多个小事务合并
  2. 使用管道（Pipeline）减少网络开销
  3. 考虑使用Lua脚本
```
