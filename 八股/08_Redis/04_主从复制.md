# 主从复制（核心面试知识）

---

## 一、主从复制概述

> Redis主从复制是指将一个Redis服务器（主库）的数据复制到其他Redis服务器（从库）

### 1.1 作用

```
1. 数据备份：从库作为主库的备份
2. 负载均衡：从库处理读请求
3. 高可用：主库故障时从库可以升级为主库
4. 水平扩展：提高读性能
```

---

## 二、主从复制原理

### 2.1 复制流程

```
┌─────────┐     RDB/命令流    ┌─────────┐
│  主库   │ ──────────────→ │  从库   │
└─────────┘                 └─────────┘

详细步骤：
1. 从库连接主库
2. 从库发送SYNC命令
3. 主库生成RDB文件
4. 主库发送RDB文件给从库
5. 从库加载RDB文件
6. 主库发送增量命令给从库
7. 从库执行增量命令
```

### 2.2 复制模式

#### 全量复制

```
- 首次连接时
- 从库断线重连后，主库偏移量不在从库的复制缓冲区范围内

步骤：
1. 从库发送SYNC命令
2. 主库执行BGSAVE生成RDB文件
3. 主库缓存期间的写命令
4. 主库发送RDB文件给从库
5. 从库清空并加载RDB文件
6. 主库发送缓存的写命令
7. 从库执行这些命令
```

#### 增量复制

```
- 从库断线重连后，主库偏移量在从库的复制缓冲区范围内

步骤：
1. 从库发送PSYNC命令（携带偏移量）
2. 主库检查偏移量是否在缓冲区
3. 主库发送增量命令
4. 从库执行这些命令
```

## 三、主从复制配置

### 3.1 主库配置

```bash
# 主库配置文件

# 绑定地址
bind 0.0.0.0

# 端口
port 6379

# 密码
requirepass your_password

# 复制相关
# 主库无需特殊配置
```

### 3.2 从库配置

```bash
# 从库配置文件

# 绑定地址
bind 0.0.0.0

# 端口
port 6380

# 密码
requirepass your_password

# 主库信息
slaveof master_ip master_port

# 主库密码
masterauth master_password

# 只读模式
slave-read-only yes
```

### 3.3 动态配置

```bash
# 从库动态设置主库
redis-cli slaveof master_ip master_port

# 从库取消主从关系
redis-cli slaveof no one

# 查看复制状态
info replication
```

## 四、复制相关配置

### 4.1 核心参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| slaveof | 主库IP和端口 | - |
| masterauth | 主库密码 | - |
| slave-read-only | 从库只读 | yes |
| repl-diskless-sync | 无盘复制 | no |
| repl-diskless-sync-delay | 无盘复制延迟 | 5 |
| repl-timeout | 复制超时时间 | 60 |
| repl-backlog-size | 复制缓冲区大小 | 1mb |
| repl-backlog-ttl | 复制缓冲区过期时间 | 3600 |
| slave-priority | 从库优先级 | 100 |

### 4.2 无盘复制

```bash
# 开启无盘复制
repl-diskless-sync yes

# 延迟时间
repl-diskless-sync-delay 5

# 优点：减少I/O，适合网络带宽大的环境
# 缺点：占用主库内存
```

### 5.1 查看复制状态

```bash

info replication

# 输出示例
```

|------|------|

## 六、主从复制问题

### 6.1 复制延迟

```
原因：
1. 网络延迟
2. 主库写操作频繁
3. 从库配置低
4. 全量复制

解决方案：
1. 优化网络
2. 增加从库
3. 提升从库配置
4. 合理设置复制参数
```

### 6.2 主库故障

```
问题：从库需要手动升级为主库

解决方案：
1. 手动执行 slaveof no one
2. 客户端切换到新主库
3. 其他从库执行 slaveof 新主库IP 端口
```

### 6.3 数据不一致

```
原因：
1. 网络中断
2. 主库配置不同
3. 从库延迟

解决方案：
1. 监控复制状态
2. 定期数据校验
3. 使用哨兵模式
```

## 七、主从复制最佳实践

### 7.1 架构设计

```
1. 一主多从：主库写，从库读
2. 树状结构：主 -> 从 -> 从（减轻主库压力）
3. 跨机房部署：提高可用性
```

### 7.2 配置建议

```bash
# 主库
maxmemory 4GB
maxmemory-policy allkeys-lru

# 从库
slave-read-only yes
repl-backlog-size 10mb
repl-timeout 30

# 监控
定期检查复制状态
```
